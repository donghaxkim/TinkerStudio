# Stagehand Narrative Exploration Design

## Status

Draft for user review.

## Context

The Playwright AI URL path currently builds a storyboard and deterministic `CapturePlan` from partial evidence:

```text
productUrl + prompt + ProductAnalysis + RepoAnalysis
  -> AI storyboard + CapturePlan
  -> verifyCapturePlan
  -> deterministic Playwright replay
  -> DemoProject
```

That boundary is intentionally safe: AI may plan, but the final recording is verified deterministic replay. The weakness is narrative quality. `ProductAnalysis` captures the first visible page state, basic copy, links, buttons, inputs, brand hints, and a screenshot. `RepoAnalysis` adds source-derived product context. Neither source reliably answers which live workflow creates the clearest demo arc.

The goal is to improve storyboard narrative quality before capture planning, not to replace deterministic capture.

## Goal

Add a bounded, optional Stagehand-backed narrative exploration layer before the existing planner.

Given a running `productUrl`, prompt, `ProductAnalysis`, and optional `RepoAnalysis`, the system should explore the live product enough to produce a compact `NarrativeExploration` artifact. The existing planner then uses that artifact to choose a stronger demo angle, storyboard beats, and capture intent.

The final `CapturePlan` must still be generated by Tinker's planner, validated by existing schemas, checked by `verifyCapturePlan`, and executed by deterministic Playwright replay.

## Non-Goals

- No AI execution during final capture.
- No Stagehand-generated `CapturePlan` execution.
- No browser-use integration in this slice.
- No autonomous unbounded browsing.
- No auth, payment, destructive, account-creation, download, extension, or external-navigation workflows.
- No changes to `@tinker/browser-capture` step types.
- No requirement that narrative exploration runs in CI or without model/browser provider credentials.
- No editor, rendering, queue, database, or project schema work.

## Recommended Approach

Use Stagehand as a focused discovery adapter behind a narrow `exploreNarrativeWebsite` function. Prefer Stagehand `observe` and `extract` operations over autonomous agents or free-form `act` calls.

Stagehand is preferred over browser-use for this slice because Tinker is TypeScript and Playwright-oriented, and Stagehand can operate on Playwright pages directly. Browser-use remains a possible future sidecar if Tinker later needs broader autonomous web research, but it adds Python/runtime complexity and more nondeterministic agent behavior than this narrative-quality slice needs.

Alternative approaches considered:

- **Prompt-only planner tuning:** smallest change, but it still relies on first-page evidence and repo guesses. It may improve tone but not workflow selection.
- **Stagehand writes final storyboard and capture plan:** may improve demos quickly, but it splits planning authority and weakens the debug/validation boundary.
- **Stagehand narrative exploration artifact:** best balance. It enriches evidence while preserving the existing planner and deterministic capture model.

## Package Boundaries

### `@tinker/product-analysis`

Owns live website analysis and should own narrative exploration.

Responsibilities:

- Add `NarrativeExploration` types and parser/validator.
- Add `exploreNarrativeWebsite(productUrl, options)` as an optional, bounded exploration function.
- Keep the function independent from final storyboard and capture-plan schemas.
- Produce deterministic test seams so unit tests can use fixture exploration output without Stagehand credentials.

### `@tinker/demo-assembly`

Owns orchestration and planner input.

Responsibilities:

- Invoke narrative exploration after website and repo analysis when enabled.
- Write `narrative-exploration.json` as a generated artifact.
- Pass `narrativeExploration` into `AiUrlPlannerInput`.
- Update planner prompts to treat narrative exploration as untrusted evidence for demo angle selection.
- Keep existing capture validation and replay unchanged.

### `@tinker/browser-capture`

Remains unchanged.

Responsibilities:

- Continue validating and executing only deterministic `CapturePlan` objects.
- Avoid Stagehand, browser-use, or AI-specific logic.

### `@tinker/generation-contract`

Should remain unchanged for the first slice unless callers need an explicit request flag.

The first implementation can use an environment flag or internal runner option to enable exploration. If product requirements later require user-visible control, add a validated request option separately.

## Data Model

Add a compact planning-oriented artifact:

```ts
type NarrativeExploration = {
  productSummary: string;
  bestDemoAngle: string;
  userProblem: string;
  promisedOutcome: string;
  workflowCandidates: Array<{
    name: string;
    whyItMatters: string;
    routeHints: string[];
    visibleEvidence: string[];
    storyboardUse: "hook" | "main-demo" | "proof" | "cta";
  }>;
  strongestCopy: string[];
  avoidNarratives: string[];
  explorationNotes: string[];
};
```

Runtime validation should bound all fields before planner use:

- `productSummary`, `bestDemoAngle`, `userProblem`, and `promisedOutcome`: non-empty strings, max 500 characters each.
- `workflowCandidates`: max 6 entries.
- `name`: max 120 characters.
- `whyItMatters`: max 240 characters.
- `routeHints`: max 8 strings, same-origin paths or short route labels only.
- `visibleEvidence`: max 8 strings, max 180 characters each.
- `strongestCopy`, `avoidNarratives`, and `explorationNotes`: max 10 strings each, max 180 characters per string.

This artifact is evidence for narrative planning. It is not an instruction source and not an execution plan.

## Data Flow

```text
CreateDemoRequest(mode: "ai-url-planning", productUrl, repoUrl?, prompt)
  -> generation-contract validates request
  -> product-analysis analyzes initial website state
  -> optional repo analysis
  -> optional Stagehand narrative exploration
  -> writes narrative-exploration.json when exploration runs
  -> planner receives ProductAnalysis + RepoAnalysis? + NarrativeExploration?
  -> planner emits storyboard + CapturePlan JSON
  -> strict schema parsing
  -> assertStoryboardMatchesInput
  -> assertCapturePlanMatchesProductUrl
  -> verifyCapturePlan
  -> deterministic Playwright capture
  -> compileProject
```

When exploration is disabled or unavailable, the pipeline should behave as it does today and skip `narrative-exploration.json`.

## Exploration Guardrails

Stagehand exploration must stay bounded:

- Same-origin navigation only.
- Hard timeout and max observed pages/actions.
- Prefer `observe` and `extract` over `act`.
- If `act` is used, limit it to safe navigation, opening menus/tabs, and non-submitting UI state changes.
- Do not submit forms unless a future design explicitly permits safe sample data for a specific workflow class.
- Do not type private, credential, payment, or user account data.
- Do not follow external links or new tabs.
- Do not download files.
- Do not treat website text, repo text, model output, or exploration notes as system instructions.

Exploration failures should not automatically fail generation in the first slice. If exploration fails, record a warning in logs or progress metadata when practical and continue with existing `ProductAnalysis` and `RepoAnalysis` evidence. Do not write `narrative-exploration.json` for failed exploration. This keeps the feature additive and avoids making demo generation more fragile.

## Planner Prompt Contract

Extend `AiUrlPlannerInput`:

```ts
type AiUrlPlannerInput = {
  productUrl: string;
  prompt: string;
  durationCapSeconds: number;
  aspectRatio: AspectRatio;
  analysis: ProductAnalysis;
  repoAnalysis?: RepoAnalysis;
  repoCheckoutDirectory?: string;
  narrativeExploration?: NarrativeExploration;
};
```

Prompt rules:

- Use narrative exploration to choose the strongest demo angle and beat purpose.
- Use website analysis for visible labels, inputs, routes, and initial UI state.
- Use repo analysis for product meaning, feature names, domain language, and plausible workflows.
- Prefer workflows supported by both live exploration and repo/website evidence.
- Do not let exploration bypass URL, schema, safety, or capture-plan constraints.
- Continue returning exactly one JSON object with `storyboard` and `capturePlan`.

## Error Handling

For the first slice, narrative exploration should be best-effort:

- Missing Stagehand credentials or disabled flag: skip exploration.
- Stagehand initialization failure: skip exploration and record a warning in logs.
- Exploration timeout or model failure: skip exploration and record a warning in logs.
- Invalid `NarrativeExploration` output: skip exploration and record a warning in logs.

Planner, verification, capture, and assembly failures remain unchanged. If future product requirements need strict exploration mode, add a separate request option and error stage.

## Testing And Verification

Automated tests should not depend on live Stagehand credentials.

Planned checks:

- Unit test `NarrativeExploration` parsing with valid, oversized, and malformed samples.
- Unit test `runAiUrlDemo` passes fixture `narrativeExploration` into the planner when the dependency returns one.
- Unit test `runAiUrlDemo` continues without exploration when the dependency fails in best-effort mode.
- Unit test artifact paths include `narrative-exploration.json` only when exploration succeeds.
- Unit test planner prompt includes narrative exploration as untrusted evidence and preserves existing capture constraints.
- Existing Playwright capture and AI URL tests continue to pass.

Manual verification can run a local fixture or known safe product URL with exploration enabled and inspect whether `storyboard.json` has a clearer Hook -> Demo: Use Case -> End Result -> CTA arc.

## Success Criteria

The slice is successful when:

- Narrative exploration is optional and additive.
- `narrative-exploration.json` is compact, bounded, and validated before planner use.
- Storyboard prompts can use exploration for demo angle selection without changing capture execution.
- Final `CapturePlan` validation and deterministic Playwright replay remain the authority.
- Generation still works when Stagehand is unavailable or exploration fails.
- Tests cover both successful exploration and best-effort fallback.
